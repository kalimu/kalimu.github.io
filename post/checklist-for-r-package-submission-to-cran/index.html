Checklist for R package submission to CRAN

21 April, 2015 programming

Version: 1.2

The last update of this post: 2018-05-10.

Below is a checklist that I wrote for my own purposes. I helps me to be sure that I did what I should have done before release a new version of R package to CRAN. Maybe it will also help others.

  1. See what new Hadley recommends:
    (http://r-pkgs.had.co.nz/release.html).
  2. Update R & packages:
    1. R
    2. all packages
    3. RStudio (stable or preview version).
  3. Update your package:
    1. Check unit tests.
    2. Update vignettes.
    3. Check reverse dependencies.
      1. If devtools functions don’t work use:
        tools::dependsOnPkgs()
    4. Import undefined globals in the NAMESPACE file
      (generated automatically by roxygen2)
  4. Update README.md file:
    1. Make change in README.Rmd.
    2. Knit the Rmd into md.
  5. Run a R CMD check locally with ‘–as-cran‘ option.
    1. If you got an error regarding qpdf missing, check:
      1. View(Sys.getenv(“PATH”) )
      2. Sys.which(Sys.getenv(“R_QPDF”, “qpdf”))
  6. Run a R CMD check on win-builder
    (devel and release versions) via ‘devtools::build_win()‘.
  7. Run a R CMD check on Ubuntu (via Travis-CI).
  8. Update ‘cran-comments.md‘:
    1. Test environments,
    2. R CMD check results (explain ERRORs, WARNINGs, NOTEs).
    3. Downstream dependencies.
    4. Resubmit comments.
  9. Update ‘Version‘ field in the DESCRIPTION file
    (major.minor.patch.dev).
  10. Update ‘Date‘ field in the DESCRIPTION file.
  11. Update NEWS.md file.
  12. Push all necessary commits to GitHub.
  13. Run again a R CMD check locally just to be sure.
    1. Use devtools::check().
  14. Push all necessary commits to GitHub.
    1. Merge branches if necessary
      Via GithHub website OR Via GIT: git merge –no-ff branch_name
  15. Check vignettes:
    1. Build your package sources
    2. Load them with:
      install.packages(“../packageName_2.0.90002.tar.gz”,
      repos = NULL, type = “source”)
    3. Use browseVignettes(“genderizeR”)
  16. Submit the package to CRAN.
    1. Use devtools::release().
    2. Confirm submission via link in the mail.
  17. Check results of submission:
    1. in all CRAN checks
      (from http://cran.r-project.org/web/packages)
    2. in comments inside the e-mail from CRAN
  18. Fix what needs to be fixed.
    1. Consider increasing version number.
    2. Check everything again.
    3. Resubmit to CRAN.
  19. After successful submission:
    1. Create a new release with tag version on GitHub repo.
      1. Copy and paste the contents of the relevant NEWS.md section into the release notes
    2. Increment the version of the package in the DESCRIPTION file (to X.X.X.9000).
    3. Promote your package
      1. Post on personal blog
      2. Post on Twitter (use #rstats)
      3. R-Bloggers
      4. etc.
  20. Update this checklist.
    1. Update the checklist version number.
